    处理隐通道 Covert Channels

    • 发现存在的隐通道后
      • 关闭隐通道 / slow it down
      • 探测试图利用隐通道的举动
      • 允许隐通道存在

    隐通道不可被完全消除，可能的应对方式如下： - 限制共享资源（硬盘/ CPU / 内存...） 只有在 相同安全层级 下会发生资源共享 - 限制带宽（引入噪声进行干扰）

    RBAC 基于角色的访问控制

    • DAC & MAC 的缺点

      • DAC 太脆了，不能满足要求
      • MAC 太严格了，不够灵活
      • 而且前两个方式的 workload 非常高

    • RBAC 的特点

      • 责任分散
      • 角色层级
      • 角色激活
      • 基于角色进行限制
    • 一些基础概念
      • Users
      • Roles
      • Object
      • Operation
      • Permission
    • RBAC 的 优点
      • 容易进行权限管理
      • 可以用于实现 DAC & MAC
      • 便于根据工作需要进行分类，如获得企业财务部门的企业融资许可，非金融部门员工可以通过财务人员的角色进行区分
      • 容易实现权限最小化，即使用户被指定为高级状态也只能在必要时拥有该权限，以防止意外发生
      • 易于任务共享，不同角色执行不同的任务
      • 易于文件分层管理，文档本身可以分为不同的角色，如信件，账单等，由用户的不同角色拥有

    Role

    “角色”是 RBAC 中的核心概念，权限取决于角色，而用户可以承担角色。 - Group：仅仅是一组用户 - Role：一组用户 + 在这个组里拥有的权限

    Security Access Control

    模型是固定的，但规则是灵活的 - Authorization management - 谁来授权？ - MAC - 基于 主客体的 security level - 主体的 level 由 sys security admin 决定 - 客体的 level 由 创建者 的 level 决定 - DAC - Centralized management - admin 一个人决定 - Hierarchical management - admin 下面还有 子admin，子 admin 真正实现 centralized manage - Ownership Management - 客体的所有者对所有访问者授权 - Collaborative management - 由共享资源的所有用户决定特定资源的访问权限 - Distributed management - 所有者可以授权别人，被授权人可以再转包 - RBAC 管理权限的降级是 RBAC 的一个重要功能，这在 DAC 和 MAC 中不存在。 - Minimize Rights - super admin 把权限分给三个子管理员 - System Administrator - 负责系统维护、用户管理、软件安装等功能 - Security Administrator - 负责安全规则配置、安全策略管理等职能 前两个人的行为会被日志记录，让审计员进行检查 - Audit Administrator - 负责检查审计记录、监控系统安全等功能 每个特权用户之间互相约束，在一类用户眼中，其他类用户就是普通用户
