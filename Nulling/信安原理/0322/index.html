    发言： - 为什么平时用 MAC 而不是数字签名呢？ - MAC 的计算比数字签名快得多（基于二进制运算） - Undeniable 服务不是一直被需要的

    Password-Based Authentication

    能不要自己写登录管理就不要自己写 - 第三方登录接口 能不要自己做就不要自己做，能不要造轮子就不要造轮子

    • 当密码从用户传送到系统时容易受到窃听
    • 密码文件总是难以 keep secret

    • 密码总是被 Guess 出来的，容易记忆的密码总是很好猜

    • 影响密码系统安全性的因素

      • 限制攻击者输入密码的频率/次数
      • 存储密码 明文/哈希
      • 互联网上传输的内容 明文/哈希
      • 修改密码的过程（如何证明是用户本人）
      • 密码的生命周期（多久要强制修改一次）
    • 一些其他因素
      • 不要把明文显示在屏幕上
      • 密码的长度和格式
      • 什么时候要求重输
      • 多次错误禁止登录

    早期密码实现 - hash

    复杂的 pattern 要求是希望用户的密码不要落到 dictionary 中

    • Stage Password 存储 hash 值而不是明文，验证 H(input) ?= H(key)
    • Unix Password 使用DES作为单向哈希函数（只用前8个char作为密钥把空串加密25次） 存储位置在 /etc/password，格式是 uname-hash(key)
      • 问题
        • 复杂度只有 \(94^8 = 6*10^{15}\) ，可以穷举
        • 大家用一些常见字串后就更少了 => 可以猜出来捏 （字典攻击 - 只对字典内容进行 hash 枚举）
    • Salting 加盐（/etc/password全局可读） 前两char纯随机salt + 一段 Hash 值（Hash(strcat(salt, pwd))） 同样的密码有 1w+ 的不同存储方式
      • 验证 Hash(strcat(salt, input)) ?= val
    • Shadow Password（/etc/shadow 仅 root 可读）
      • /etc/password 仅存储用户名

    Guessing & Protection

    • Guessing
      • 猜 默认 / 短密码 / 有意义单词
      • 尝试用户私人信息：电话号码、身份证、房间号
      • 使用木马监听键盘输入
      • 使用一些 social engineering 技术
    • Protection
      • char + num / random
      • 木马防护：使用安全的工具进行登录
      • 不同网站用不同密码
      • 不要相信第三方软件
      • 不要在一些迷幻的地方登录自己的账号

    Bio / Behavior Authentication

    • Bio：指纹、虹膜、视网膜
    • Behavior：笔记、姿态、打字频率
    • 优点：不会丢、不会忘
    • 缺点：
      • 设备安装、维护需要成本
      • 对比算法的准确性：假阴性和假阳性
      • 隐私问题：指纹造假
    • 容错率（不可避免）
      • False Accept：接收恶意用户
      • False Reject：拒绝正常用户
      • 当 false accept == false reject 时，称为 equal error rate
    • 一些考量
      • Universality 每个人都有这个特征吗
      • Uniqueness 每个人的特征不一样吗
      • Permanence 随着时间流逝，这个东西还能用吗
      • Collectablility 容易获得吗
      • Performance 计算的速度、效率、鲁棒性
      • Acceptablilty 大家愿意用吗
      • Circumvention 可以绕过这个系统吗
