    8 IO

    1 Intro

    Privilege 和 Priority 的概念是正交的，两者完全不相关！

    • Privilege 权限

      不是所有程序都有权限执行 所有类型的指令，我们将所有程序划分为 Privileged / UNprivileged 两类，前者运行在 Supervisor Mode，后者运行在 User Mode

      只有运行在 Supervisor Mode 的程序可以运行 所有类型指令，并访问 任意内存空间

      在 LC-3 中可能导致的 Exception 有：

      • Privilege Mode Exception：越权执行指令，比如在用户态执行 RTI

      • Access Control Violation Exception：越权访问内存地址，比如在用户态尝试访问 System Space 或 I/O Page

    • Priority 优先级

      每个程序都被赋予了 Priority，这是时间片分配的依据

      LC-3 共支持 8 个优先级，其中 Level 7 最高，Level 0 最低

      一些策略允许高优先级程序抢占低优先级程序的时间片

    • PSR 程序状态寄存器

      Processor Status Register，被映射到 xFFFC

      PSR 记录了 当前程序 的 Privilege & Priority 信息

      Priv useless Prio useless condition desc
      1 - - - - xxx - - - - - N Z P Supervisor
      0 - - - - xxx - - - - - N Z P User

    • LC-3 内存结构

      LC-3 采用 16-bit address，其内存空间为 x0000 ~ xFFFF

      • System Space x0000 ~ x2FFFF

        存放了 OS 必要的数据结构和代码，要求 Supervisor Privilege

      • User Space x3000 ~ xFDFF

        所有程序均可访问

      • I/O Page xFE00 ~ xFFFF

        用于表示参与 IO 功能的寄存器 & 一些处理器相关的特殊寄存器：addr(PSR) = xFFFC, addr(MCR) = xFFFE

        需要 Supervisor Privilege（但这一部分已经不用做内存了orz）

      • SSP / USP

        指向 Supervisor / User Stack 的栈顶指针

        由于程序在任意时刻只能运行在 一种状态下，因此两个栈不会同时活跃

        未使用的指针会被存放在 reg[Saved_SSP] / reg[Saved_USP] 中

    2 Basic Characters

    2.1 Memory-Mapped & Special Instructions

    通过指令访问 IO 设备寄存器时，必须 指定目标寄存器。这通常有两种实现方式：

    1. （少数）使用 special IO 指令：为读写设备寄存器单独分配 opcode

    2. （多数）使用和操作 memory 相同的 data movement 指令，也被称为 “内存映射” Memory Mapped

      • 这种实现更加灵活，能够通过指令的目标地址进行判断：

        load 目标地址为 device register 时，就是一条 input 指令

        store 目标地址为 device register 时，就是一条 output 指令

      • 这也要求 device register 拥有与 Memory 相同格式的标识

        => 这正是上图 I/O Page 段实现的功能：将 I/O device registers 映射到一组地址

    2.2 Asynchronous & Synchronous

    与 CPU 的运算速度相比，IO 设备的速度是非常缓慢的。

    • 这一矛盾可以通过硬件设计解决 Synchronous

      Fix rate with CPU clock

      • 让硬件以一个 固定的低速率 读取用户输入

        但这也要求打字员的输入频率 始终与时钟周期同步（进行同步交互 ） => 这显然不科学

      • 或者让硬件 定期读取 device register

    • 异步操作 Asynchronous

      Varying rate with CPU clock

      问题的主要矛盾在于用户输入与处理器工作速度不一致，是异步的。

      => 我们需要通过 protocol / handshake mechanism 来控制信息的发送与接收。

      • 使用 Flag 是最简单实现方式，例如使用 1-bit（ ready bit ）指示键盘的输入状态：

        用户每输入一个字符，令 ready bit = 1

        CPU 每处理一个字符，令 ready bit = 0（flag == 0 时不会执行读取操作）

    2.2 Interrupt-Driven & Polling

    中断驱动（ Interrupt Driven）与 轮询（ Polling ）是 IO设备与 CPU 进行交互的两种基本方式，本质区别在于 谁控制着交互：

    • 中断驱动（ IO 控制）

      CPU 可以干自己的事情，当输入数据准备完毕后，IO 设备会 主动通知 CPU 去 input device register 中读取数据

      • 会强行中止当前正在运行的程序（处理结束后恢复）

      • 不需要循环检测 ready bit

    • 轮询（ CPU 控制）

      CPU 必须不断读取 ready bit 并检查判断 ?flag == 1，符合条件时再从 input device register 中读取输入数据

    3 Input from Keyboard

    3.1 KBDR & KBSR

    从键盘读取输入需要两类信息：

    1. Data Register：存储了输入数据的 ASCII
    2. Synchronization Mechanism：让处理器知道是否待处理数据

    我们使用两个寄存器存储上述信息：

    • 数据寄存器 KBDR addr = xFE02

      15-8 ( 8 bit ) 7-0 ( 8 bit )
      useless ASCII

    • 状态寄存器 KBSR addr = xFE00

      15 14-0 desc
      Ready Bit useless
      1 useless Ready
      0 useless Empty

    3.2 基本实现

    我们可以从上一节提供的信息中得出：KBSR[15] = ready bit，KBDR[7:0] = ASCII(input)

    • 输入：按键对应的 ASCII 被存入 KDSR[7:0]，键盘硬件电路自动令 KBSR[15] = 1

      仅当 flag == 0 时允许新的输入

    • 处理：键盘硬件电路自动令 KBSR[15] = 0

    在 Polling 交互下，CPU 将反复读取并检验 flag：

    • 当 flag == 1 时，CPU 便将 KBDR 中的 ASCII 值拷贝至 处理器内部的本地寄存器 中
    • flag 实现的同步机制确保输入有且仅有一次读取机会：
      • 输入不会被重复读取：flag 会在拷贝后自动 clear
      • 输入不会丢失：键盘在上一个输入被处理前禁用
    拷贝 ASCII 至 R0 后，返回执行下一个任务
    START: ;                 loop: ? flag == 1
     . LDI R1, A;        R1 = KBSR = [flag]xxxxxxx
       BRzp START;       if (flag == 0)  continue
       ;
       LDI R0, B;        R0 = KBDR = asc(input)
       BRnzp NEXT_TASK;  执行其他任务

A:     .FILL xFE00;     mapped_addr(KBSR)
B:     .FILL xFE02;     mapped_addr(KBDR)

    3.3 Memory-Mapped 实现

    在前面的内容中，我们已经学习了 load 指令在 EXECUTE 阶段的三个基本步骤：

    1. MAR = address
    2. 访问内存，令 MDR = M[MAR] = M[address]
    3. 拷贝数据，令 reg[dst] = MDR = M[address]

    在 “内存映射” 方式下，操作步骤 完全一致（映射后的寄存器具有 16 bit 地址），只不过：

    • MAR 中保存的是 设备寄存器映射地址，而非内存地址（ IO Page 不算内存）
    • 读取的数据来自于 device register，而非 Memory

    3.4 输入提示 & echo

    本节的实现添加了两个功能：

    1. 输入提示

      首先输出一个 \n，强调当前程序输出

      • OS 产生的输入提示信息以 % or C: 开始
      • 编辑器产生的输入提示信息以 : 开始

    2. Echo

      当用户输入字符后立刻进行 echo

      用户输入结束后，额外打印一个 \n

    一个更完备的输入处理程序
    # input = 1 * char

START:      ;                Callee-Store
            ST R1, SaveR1;
            ST R2, SaveR2;
            ST R3, SaveR3;
            ;
            LD R2, Newline;  R2 = asc('\n')
L1:         ;                输出起始换行符
            LDI R3, DSR;
            BRzp L1;
            STI R2, DDR;     
            ;
            LEA R1, Prompt;  R1 = &prompt[输出提示]
LOOP:       LDR R0, R1, #0;  R0 = prompt[i]
            Brz INPUT;       len(prompt[i] == '\0') 准备 echo
L2:         ；               循环输出 str(prompt)
            LDI R3, DSR;
            BRzp L2;
            STI R0, DDR;
            ADD R1, R1, #1;
            BRnzp LOOP;
INPUT:      ;                loop: echo(input)
            LDI R3, KBSR;   
            BRzp INPUT;
            LDI R0, KBDR;    R0 = asc(input)
L3:         LDI R3, DSR;     
            BRzp L3;
            STI R0, DDR;     echo(input)
L4:         ;
            LDI R3, DSR;
            BRzp L4;
            STI R2, DDR;     输出结束换行符
            ;                Callee-Restore
            LD R1, SaveR1;
            LD R2, SaveR2;
            LD R3, SaveR3;
            BRnzp NEXT_TASK;

SaveR1:     .BLKW 1
SvaeR2:     .BLKW 1
SaveR3:     .BLKW 1
DSR:        .FILL 0xFF04
DDR:        .FILL 0xFF06
KBSR:       .FILL 0xFF00
KBDR:       .FILL 0xFF02
Newline:    .FILL 0x000A;               asc(`\n`)
Prompt:     .STRINGZ "Input a char >"   输入提示字符串

    4 Output to Monitor

    4.1 DDR & DSR

    类似于键盘输入，向显示器输出同样需要两个寄存器记录相关状态：

    • 输出数据寄存器 DDR addr = xFE06

      15-8 ( 8 bit ) 7-0 ( 8 bit )
      useless ASCII

    • 状态寄存器 DSR addr = xFE04

      15 14-0 desc
      Ready Bit useless
      1 useless Ready
      0 useless Busy

    4.2 基本实现

    1. 循环测试 ? DSR[15] == 1

    2. 符合条件时令 DDR[7:0] = asc(output)，并启动屏幕显示

    3. 显示完成后，硬件电路将自动令 DSR[15] = 0，程序执行其他任务

    将数据写入 DSR，返回执行下一个任务
    START: ;                 loop: ? flag == 1
     . LDI R1, A;        R1 = DSR = [flag]xxxxxxx
       BRzp START;       if (flag == 0)  continue
       ;
       STI R0, B;        DDR = R0 = asc(output)
       BRnzp NEXT_TASK;  执行其他任务

A:     .FILL xFE04;     mapped_addr(DSR)
B:     .FILL xFE06;     mapped_addr(DDR)

    4.3 Memory-Mapped 实现

    数据通路中增加了：

    • 输入源：DSR，用于 flag 检测
    • 输出源：DDR，用于记录待输出数据

    融合 - echo 实现
    START:  LDI R1, KBSR; 检查是否有输入
        BRzp START;   if (flag == 1) continue
        LDI R0, KBDR; R0 = asc(input)
ECHO:   LDI R1, DSR;  检查屏幕是否空闲
        BRzp ECHO;    if (flag == 1) continue
        STI R0, DDR;  DDR = asc(input)
        BRnzp NEXT_TASK; 

KBSR:   .FILL xFE00;
KBDR:   .FILL XFE02;
DSR:    .FILL xFE04;
DDR:    .FILL xFE06;

    5 更完备的 IO 实现

    下图的数据通路糅合了 3.3 & 4.3，并添加了一些额外功能：

    • 添加了对 Interrupt-Driven IO 的支持：

      令 KBSR & DSR 不仅可读，而且 可写

    • 基于以下 3 个输入信号，地址控制逻辑可能采取不同的操作：

      1. MAR

        存在两种可能取值 - 内存地址 - 设备寄存器的内存映射地址

      2. 控制信号 MIO.EN

        表明本周期需要处理的 data movement 类型：from/to Mem OR Input/Output

        MIO.EN = 0 表示本周期不进行数据移动

      3. 控制信号 R.W

        表明执行的是 load / store，进行 load 操作时会通过 INMUX 选取正确的数据源

    6 考虑中断的 Polling

    在本章中，我们通过如下的循环结构不断对 ready bit 进行判断：

    1
2
3
4
    Loop:   LDI R1, DSR
        # 中断可能发生在这个位置
        BRzp Loop
        STI R0, DDR

    • 中断可能发生在 LDI 与 BRzp 之间，而在中断处理过程中，DDR 的就绪状态可能已经发生了变化：

      若 DDR 由 可用 -> 不可用，就会因 BR 判断信息之后导致不能成功写入信息（少写）

    一个 naive 的解决方案是在轮询时禁用所有中断（这在需要长时间轮训的情形下极不理想）

    • 更好的解决方案是在每次执行指令序列 LDI -> BRzp -> STI 时禁用中断，随后再次被启用

      在这种情况下，中断不会干扰序列 LDI -> BRzp -> STI ，同时也无需等待整个轮询过程完成

    类似于 加锁/解锁 语句应该紧贴互斥操作的思想
