9 陷入 & 中断

在上一章中，我们为硬件设备分配了 数据 / 状态寄存器 以实现 IO，并在应用程序中实现了 IO 的同步机制。

  • 然而在实际的开发过程中，程序员往往不需要知道 IO 的具体实现细节 —— 调用库就完事了！

  • 此外，允许应用程序直接访问 KBDR 和 KBSR 也是不合理的：

    • 由于各个进程对 IO 设备的访问是互斥的，一个进程搞砸就会霍霍一连串的其他进程

    • 这也是 hardware register 的地址被设置在 Privileged Memory Address 中的原因

=> 使用 TRAP 指令摇人处理 IO 似乎是更加简单的方案

毕竟 OS 一定具有 Supervisor Privilege，而且别人写好的东西看起来更香

这种摇人干活的行为被称为 “系统调用” 或 “服务调用” System / Service Call

1 陷入 Trap

Trap 机制的实现离不开以下几个要素：

  1. 一系列的 Service Routine（系统函数）

    虽然这些函数是由 OS 提供的（LC-3 给大家塞了 256 种），但可以由应用程序 发起调用

    这些系统函数实际上运行在 内核态

  2. 起始地址列表

    有时也被称为 “系统控制块” System Control Block，或 “陷入矢量表” Trap Vector Table

    我们之前提到过 TRAP 指令通过 trapVector 指定调用的服务类型，而该表记录了 trapVector -> 各服务程序起始位置 的映射

    在 LC-3 中，该表位于 x0000 ~ x00FF，记录了所有 256 个服务程序的相关信息

  3. TRAP 指令

    允许用户程序调用 内核态 下的指定服务，并在执行完成后取回控制权

  4. Linkage

    服务程序完毕后自动返回用户程序的机制

1.1 TRAP 指令

在真正执行 service routine 之前，TRAP 指令完成了两件事：

  • 根据 trapVector 令 PC 指向对应 service routine 的起始位置（在起始地址列表中查找）

  • Linkage：记录返回地址，提供从 service routine 返回断点继续执行的机制

TRAP 指令的基本格式如下：

Opcode useless TrapVector
1111 0000 xxxx xxxx

在指令周期的 EXECUTE 阶段，TRAP 指令完成了以下 3 件任务：

程序状态寄存器 PSR 记录了当前进程的 Priority & Privilege 信息；寄存器 R6 指向当前进程使用的栈（ System / User Stack）。

  1. 将 PSR & PC 的值压入 System Stack

    • Linkage：由于 PC 在 FETCH 阶段就已经自增，此时恰好指向了返回地址

    • 执行 TRAP 指令的程序可以运行在 用户态/内核态

      由于 用户态进程 的栈指针 R6 指向了 User Stack，我们必须对其进行切换：Saved_USP = R6; R6 = Saved_SSP

  2. 修改 PSR

    • PSR[15] = 0 ：执行 Service Routine 需要 Supervisor Privilege
    • PSR[10:8]不变：Callee（Service Routine）与 Caller 具有 相同的 Priority

  3. 修改 PC

    对 trapVector 进行 零扩展，随后取出起始地址列表中的对应表项（指定服务函数的起始位置）交给 PC

下面是一个简单的例子：将用户所有的输入转为小写显示，用户输入 7 则退出程序

1.2 RTI 指令

全称为 Return from Trap or Interrupt

RTI 指令用于从 系统服务函数/中断处理函数 返回原应用程序中的断点，基本结构如下：

Opcode useless
1000 0 * 12 bit

  1. 鉴于 TRAP 指令在 EXECUTE 阶段将 PSR & PC（返回地址）压入 System Stack，此处从 System Stack 的栈顶弹出两个数即可完成恢复

  2. 由于原进程可能运行在 用户态/内核态，我们需要对恢复后的 PSR 进行检查

    若 PSR[15] == 1，则表明原程序运行在用户态 => 则需要对 R6 进行恢复：Saved_SSP = R6; R6 = Saved_USP

1.4.2 HALT 中断处理程序

我们在前面介绍过如何控制计算机的运行与停止 —— 这实际上是通过控制 timer 实现的：

\[ clock = RUN\_latch \ \& \ output(晶体振荡器) \]
  • 晶体振荡器 crystal oscillator 负责输出一个周期恒定的方波信号
  • 我们通过令 RUN latch = 0 来停止 clock 的工作

  • 早期的 ISA 会单独设置一个 HALT 指令以停止时钟，但鉴于 HALT 指令不太常用，让其占用单独的 opcode 就比较浪费

    => 在现代 ISA 中，我们通过 TRAP 实现 HALT 操作

  • 在 LC-3 中 MCR[15] = RUN latch，下面是 Halt 中断处理的一个实现：

    MCR 寄存器的全称是 Machine Control Register，位于 xFFFE

    1. 输出提示 "Halting the machine"
    2. 通过 MCR & (0111 1111 1111 1111) 令 MCR[15] = 0，而其余位不变

         RTI

2 中断 Interrupt

2.1 实现 Interrupt-Driven IO

  • Interrupt-Driven IO 的三个阶段：

    1. 强制中止当前正在运行的进程
    2. 执行 IO 设备需要的中断处理程序
    3. 返回执行之前运行的进程

  • 为了实现上述功能，中断驱动 IO 需要以下两种机制：

    1. 中断触发：The mechanism that enables an I/O device to interrupt the processor
    2. 中断处理：The mechanism that handles the interrupt request

2.1.1 中断触发 Cause an Interrupt

中断处理进程也可能被 具有更高优先级 的其他中断处理进程打断

下面是 IO 设备被允许中断当前进程的三个 必要条件：

  1. The IO device must want service

    一般来说就是 KBSR / DSR 中的 ready bit = 1

  2. The device must have the right to request the service

    类似的，在 KBSR / DSR 中存在由 OS 操作的 interruptenable bit(IE)

    在 LC-3 中是 bit[14]

    IE = 1 表示当前设备有权要求处理器提供服务

  3. The device request must be more urgent than what the processor is currently doing

    LC-3 中具有 PL0 ~ PL7 8 种不同优先级，且数字越大优先级越高

只有当上述条件均被满足时，处理器才会暂停正在处理的进程，转而处理 IO 中断请求

INT Signal

INT 信号其实就是上述三个条件的综合表征结果：

\[ INT = bit[15]\ \&\ bit[14]\ \& \ PL(device)>PL(curProcess) \]

处理时机

  • 中断可能发生在指令周期的任何阶段，如果我们 立刻停止手上的工作，那么：

    对于 FETCH / OPERAND 阶段：我们需要记录当前指令已完成的工作成果 + to do

  • 仅允许在 instruction boundary 处理中断似乎更加合理：我们只需要记录返回地址，而不用担心指令执行的原子性

    一个简单的实现是：在每个指令周期结束时，检测是否有待处理的中断请求

2.1.2 中断处理 Handle Request

Initiate

本阶段需要完成两个任务：

  1. 保存被打断进程的状态

    在 Supervisor Stack 中保存 PSR 和 PC（所有通用寄存器由中断处理程序进行 Callee-Save）

    若被打断程序为用户态进程，我们还需要更改 R6(sp) 使其指向 Supervisor Stack：

    Saved_USP = R6; R6 = Saved_SSP

  2. 加载中断处理程序的状态

    • PC 将被修改为 InterruptTable[INTV]

      • 产生中断的 IO 设备会向处理器发送：8-bit 中断矢量 INTV + 中断请求信号 + 设备优先级

      • 当同时有多个设备请求中断处理时，将选择具有 最高优先级 的 INTV 并将其扩展为 16-bit

    类似于 TRAP 矢量表，Interrupt 矢量表位于内存的 x0100 ~ x01FF，实现了 中断矢量 INTV -> 中断处理程序起始地址 的映射

    • PSR 将根据以下规则被修改

      • PSR[15] = 0 中断处理程序运行在内核态
      • PSR[2:0] = 010 暂时没有执行中断处理程序中的指令，统一初始化
      • PSR[10:8] = Priority(request) 中断处理程序优先级与发起请求的设备一致
Service

在上一步中，我们已经使 PC 指向了对应中断处理程序的起始位置，顺序执行即可

Return

中断处理程序的最后一条指令一定是 RTI，此时：

  • PSR & PC 被从 Supervisor Stack 顶部弹出并恢复

    PSR[15] & PSR[10:8] 已经与中断前状态一致，不需要单独修改

  • 若原进程运行在用户态，我们同样需要对 R6 sp 进行恢复：Saved_SSP = R6; R6 = Saved_USP
